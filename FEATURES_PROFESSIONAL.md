### 2. **SMTP Mailbox Verification**

**Real inbox checking without storing emails**

- Connects directly to mail servers via SMTP protocol
- Verifies if mailbox actually exists using RCPT TO command
- No emails are sent or stored on server
- Results: `exists: true/false/unknown`

**How it works:**

1. Gets MX records for the domain
2. Connects to highest priority MX server
3. Performs SMTP handshake (EHLO → MAIL FROM → RCPT TO)
4. Returns whether mailbox exists based on server response
5. Connection is closed immediately (QUIT)

**Limitations:**

- Some mail servers block external SMTP connections
- Corporate domains may always return positive (catch-all)
- Rate limited to 20 checks per minute
- Should be used sparingly to avoid blacklisting

**API Endpoint:**

```bash
POST /api/smtp-verify
Content-Type: application/json
X-API-Key: your-api-key

{
  "email": "test@example.com"
}

Response:
{
  "email": "test@example.com",
  "domain": "example.com",
  "mxHost": "mx1.example.com",
  "exists": true,
  "smtpCode": 250,
  "message": "Mailbox exists",
  "warning": "SMTP verification may not be 100% accurate",
  "timestamp": "2026-03-12T..."
}
```

## 📈 Usage Recommendations

### When to use SMTP verification

✅ **Good uses:**

- Final verification of critical emails (leads, customers)
- Checking emails that failed DNS validation
- One-time verification of important contacts
- Suspicious or edge case emails

❌ **Avoid:**

- Bulk validating entire lists via SMTP
- Repeatedly checking same emails
- Using as first validation step
- Free email providers (Gmail, Yahoo) - they block it

### Best practice workflow

1. **Syntax check** (client-side, instant)
2. **DNS validation** (server, fast)
3. **WHOIS/Website check** (server, adds context)
4. **SMTP verification** (server, use sparingly for unknowns)

### Bulk processing strategy

- Split large lists into 100-email batches
- Run batches sequentially (not parallel)
- Add delays between batches
- Skip SMTP for bulk operations
- Use Standard validation level for speed

## 🚨 Monitoring & Troubleshooting

### Health Check

```bash
GET https://your-app.onrender.com/health

Response:
{
  "status": "ok",
  "privacy": "no data stored",
  "timestamp": "2026-03-12T..."
}
```

### Common Issues

**"Unauthorized" error:**

- API key not set or incorrect
- Go to Settings → Enter API key from Render dashboard

**"Rate limit exceeded":**

- Too many requests in 1 minute
- Wait 60 seconds and try again
- For SMTP: max 20/minute, for others: max 100/minute

**"Connection timeout" on SMTP:**

- Mail server blocking external connections
- Normal for many corporate servers
- Result will show `exists: "unknown"`

**Cold start delay:**

- App sleeping on free tier
- First request after 15 min takes ~30 sec
- Upgrade to paid plan for always-on

### Logs & Debugging

- Access logs in Render dashboard → Logs tab
- Filter by error level
- Check server startup messages
- Monitor request patterns
